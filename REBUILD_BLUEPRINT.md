# Chronos v2 — Rebuild Blueprint

Supersedes the sequential MVP described in the original SPEC.md. This
version implements the multi-agent orchestrator, self-healing loop, and
Bob native-mode alignment specified by the project lead.

## 1. Core Architecture — Three Coordinated Agents

Agents run in **separate Bob context windows** (not one shared conversation),
coordinated through a shared state directory rather than shared chat
context. This is the real mechanism for "parallel sub-agents" inside
Bob's actual IDE architecture — each agent polls for its required input
file(s) and writes its own output file(s), so Modernizer and Compliance
Officer can each start as soon as their upstream dependency exists rather
than waiting on a single linear script.

```
chronos/state/
├── 00-lock.json              # simple state flags: cartographer_done, modernizer_done, compliance_done
├── dependency-map.json       # Cartographer output
├── changelog.md              # Modernizer output (append-only, per fix)
├── build-log.txt             # Modernizer's compiler/test output, latest run
├── compliance-report.json    # Compliance Officer output
└── pr-draft.md                # Final PR description
```

### Agent 1 — Cartographer (Analysis)
- Mode: `cartographer` (read-only tools)
- Trigger: run first, no dependency
- Job: recursively scan `legacy-repo/`, build dependency map, flag deprecated patterns
- Writes: `state/dependency-map.json`, sets `cartographer_done: true` in `00-lock.json`

### Agent 2 — Modernizer (Refactoring)
- Mode: `modernizer` (read + write + terminal execution)
- Trigger: polls `00-lock.json` for `cartographer_done: true`
- Job: refactor deprecated methods per the dependency map, swap outdated library versions, run the test suite / compiler after each change
- Writes: `state/changelog.md` (append per change), `state/build-log.txt` (latest run output), sets `modernizer_done: true` when the suite passes clean

### Agent 3 — Compliance Officer (SecOps Audit)
- Mode: `compliance-officer` (read + limited write to `state/` only)
- Trigger: polls `00-lock.json` for `modernizer_done: true`
- Job: static analysis for vulnerabilities, hardcoded secrets, and regulatory checks (GDPR/HIPAA rule set, section 4 of original SPEC.md) against the modernized code
- Writes: `state/compliance-report.json`, sets `compliance_done: true`

## 2. Self-Healing Compile-Fail-Fix Loop (Modernizer only)

This lives entirely inside the Modernizer agent's own working loop —
it is not a separate agent.

```
LOOP (max 5 attempts per file):
  1. Modernizer applies a refactor to a file
  2. Modernizer runs the build/test command in the terminal
     (e.g. `mvn -q -f legacy-repo/pom.xml test-compile` and, if a test
     suite exists, `mvn -q -f legacy-repo/pom.xml test`)
  3. IF the command exits non-zero:
     a. Capture full stdout/stderr into state/build-log.txt
     b. Feed that error log back into the SAME Modernizer context
        as the next instruction: "This change failed to compile/test.
        Here is the error log: <log>. Fix the specific issue without
        reverting the modernization goal, then re-run the build."
     c. Increment attempt counter, go to step 1
  4. IF the command exits 0 (clean): log success to changelog.md,
     move to the next flagged issue
  5. IF max attempts reached without a clean pass: STOP, mark the file
     as "needs manual review" in changelog.md, do NOT silently revert
     and do NOT proceed to sign off compliance on that file
```

Important constraint: the loop must never claim success without a real
passing build. A capped attempt count with an honest "needs manual
review" fallback is required — an agent that quietly reverts a failed
change and reports success anyway is a demo-breaking failure mode.

## 3. Bob Native Mode Alignment

| Bob Mode | Chronos Function |
|---|---|
| **Ask** | User queries repo state; invokes Cartographer's read-only analysis and returns a summary — no writes |
| **Plan** | Produces the structured manifest — a Plan-mode session reads `dependency-map.json` and outputs a concrete file-by-file modification list with target framework versions, which becomes the input Modernizer executes against |
| **Agent** | Executes the full pipeline: Modernizer's refactor + self-healing loop, then Compliance Officer's audit, ending in a drafted PR description (`state/pr-draft.md`) summarizing all changes and compliance findings for human review |

Note: `AGENTS.md` remains Bob's own project-context file generated by
`/init` — it should NOT be repurposed as the modification manifest. The
Plan-mode manifest gets its own file: `state/plan-manifest.md`, to avoid
conflicting with Bob's native use of `AGENTS.md`.

## 4. Separation of Concerns — What Changed From v1

- Removed: any single "run everything" prompt across all three agents in one context
- Removed: hardcoded reliance on one linear conversation thread
- Added: shared state directory as the coordination layer between agents
- Added: self-healing loop, scoped strictly to Modernizer, with a hard attempt cap and honest failure reporting
- Added: explicit Ask/Plan/Agent mode hooks, not just three custom modes run back to back
- Unchanged: the compliance rule set, the seeded legacy-repo/ demo target, the dashboard's role as a read-only viewer of `state/`

## 5. v3 — Multi-Target, Language-Agnostic Pipeline with GitHub PR Automation

v3 generalizes the architecture to support multiple target languages. The coordination
mechanism, lock file protocol, and self-healing loop are unchanged — the per-target
state directory structure is the only structural change.

### Per-Target State Directories

Each target gets its own isolated state subdirectory:

```
chronos/state/
├── java/        # Java 11 clinic records — all pipeline outputs present
│   ├── 00-lock.json              # cartographer_done, modernizer_done, compliance_done: true
│   ├── dependency-map.json
│   ├── architecture-summary.md
│   ├── changelog.md
│   ├── build-log.txt
│   ├── compliance-report.json
│   └── pr-draft.md
└── python/      # Python 2 clinic records — Cartographer not yet run
    └── 00-lock.json              # all flags: false
```

### Second Target — legacy-repo-python/

A Python 2 clinic records app mirroring `legacy-repo/` in structure and seeded issues.

**Seeded legacy patterns (Modernizer targets):**
- `print` statement → `print()` function
- `xrange()` → `range()`
- `dict.iteritems()` → `.items()`
- `except Exception, e:` → `except Exception as e:`
- `import urllib2` → `import urllib.request`
- Implicit integer division (`days_lived / 365.25`) → explicit date arithmetic for age calculation

**Seeded compliance issues (Compliance Officer targets — identical rule set):**
- SEC-001: hardcoded `DB_PASSWORD` and `NOTIFICATION_API_KEY` in `config.py`
- GDPR-032 / HIPAA-164: `ssn` and `diagnosis` as plain unencrypted strings in `patient.py`
- GDPR-005: `audit_logger.py` logs full patient record (incl SSN) to stdout on every access
- SEC-002: `hashlib.md5()` applied bare/unsalted to SSN in `patient_repository.py`

### Language-Agnostic Mode Instructions

All three custom modes now detect the target language from file extensions and apply
the appropriate rule set and build gate:

| Language | Build gate | Tool check |
|---|---|---|
| Java (`.java`) | `mvn -q -f <target>/pom.xml test` | `mvn -version` |
| Python (`.py`) | `python3 -m pytest <target>/tests/ -q` | `python3 -m pytest --version` |

The compliance rule set (SEC-001, GDPR-032, GDPR-005, HIPAA-164, SEC-002) is semantic,
not syntactic — it applies identically to Java and Python implementations.

### Python Test Suite

5 pytest tests in `legacy-repo-python/tests/`, equivalent to the Java JUnit suite:

| Test | Equivalent Java test |
|---|---|
| `test_get_age_in_years_returns_correct_whole_years` | `PatientTest.getAgeInYears_returnsCorrectWholeYears` |
| `test_patient_id_is_stored_as_integer` | `PatientTest.constructor_parsesIdStringToInteger` |
| `test_hash_for_lookup_is_deterministic` | `PatientRepositoryTest.hashForLookup_isDeterministic` |
| `test_hash_for_lookup_differs_for_different_inputs` | `PatientRepositoryTest.hashForLookup_producesDifferentOutputForDifferentInput` |
| `test_get_all_patients_returns_correct_count` | `PatientServiceTest.getAllPatients_returnsSameCountAsRepository` |

Tests fail against the legacy Python 2 code (SyntaxErrors at collection time) and pass
after modernization — same dynamic as the Java suite.

### Dashboard Target Selector

A Java / Python toggle in the dashboard header switches `OUTPUTS()` between
`../state/java/` and `../state/python/`. All three render functions are unchanged;
the Before/After subtitle updates to reflect the active target's gate tool (`mvn test` vs `pytest`).

### GitHub PR Automation (End of Agent Pipeline)

After all three pipeline flags are `true` in `state/<lang>/00-lock.json`, the Modernizer
executes the PR automation step using `git` and `gh` CLI commands already available in
the `execute` group. No new tool permissions are required.

**Full sequence:**

```
1. INIT (one-time, skip if repo_initialized: true in lock file)
   git remote add origin https://github.com/TECXBOY/Chronos.git
   git add .
   git commit -m "chore: initial Chronos v3 baseline"
   git push -u origin main
   → writes repo_initialized: true to state/java/00-lock.json AND state/python/00-lock.json

2. BRANCH
   BRANCH=chronos/modernize-<lang>-<timestamp>
   git checkout -b $BRANCH

3. STAGE (scoped — never git add . post-init)
   Java:   git add legacy-repo/ state/java/
   Python: git add legacy-repo-python/ state/python/
   git status --short   ← shown to user before commit

4. COMMIT (message built from real state data)
   git commit -m "feat(chronos-<lang>): modernize <lang> clinic records
   — N changes, N files. Gate: <tool> passing. Compliance: Nh/Mm/Ll"

5. PUSH BRANCH
   git push origin $BRANCH

6. BUILD PR BODY → state/<lang>/pr-body.md
   Real content from state/<lang>/changelog.md + compliance-report.json:
   - §1 Modernization Changes (verbatim changelog)
   - §2 Compliance Findings (Markdown table from JSON)
   - §3 Draft PR checklist (human review required)

7. CREATE DRAFT PR
   gh pr create --draft --base main --head $BRANCH \
     --title "Chronos: automated modernization + compliance audit — <LANG>" \
     --body-file state/<lang>/pr-body.md
   → returns PR URL

8. FINALIZE
   Write pr_url to state/<lang>/00-lock.json
   Report PR URL to user
```

**Safety guarantees:**
- `--draft` flag always present — human must promote to ready-for-review manually
- Never `git push origin main` after init, never `git push --force`
- Never `gh pr merge` or `gh pr edit --ready-for-review`
- Staged files always scoped to the active target directory only
- Auth reuses existing `gh` keyring (TECXBOY, `repo` scope) — no new credentials

**`00-lock.json` new fields (both targets):**
```json
{
  "repo_initialized": false,
  "repo_remote": "https://github.com/TECXBOY/Chronos.git",
  "pr_url": null
}
```
